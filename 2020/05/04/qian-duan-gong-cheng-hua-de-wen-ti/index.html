工程化

前端工程化目的：提升业务效率

为什么前端要工程化？为了提高前端开发效率，提高前端应用的可扩展性、可维护性等性能。

前端工程化的内容
  1. 规范：代码规范、目录结构规范、前后端接口规范、文档规范、commit规范、流程规范等
  2. 分支管理：不同的开发人员开发不同的功能或组件，按照统一的流程合并到主干
  3. 构建：主干更新后，自动将代码编译为最终的目标格式，并且准备好各种静态资源
    1. 编译： babel
    2. 打包： webpack
  4. 部署：将构建好的代码部署到生产环境
  5. 版本管理，发布方式（纯前端发布）
大型网站的一些问题？
  • Facebook整站有10000+个静态资源；
  • 每个静态资源都有可能被翻译成超过100种语言版本；
  • 每种资源又会针对浏览器生成3种不同的版本；
  • 要针对不同带宽的用户做5种不同的打包方法；
  • 有3、4个不同的用户组，用于小批次体验新的产品功能；
  • 还要考虑不同的送达方法，可以直接送达，或者通过iframe的方式提升资源并行加载的速度；
  • 静态资源的压缩和非压缩状态可切换，用于调试和定位线上问题
  • 高性能：CDN部署、缓存控制、文件指纹、缓存复用、请求合并、按需加载、同步/异步加载、移动端首屏CSS内嵌、HTTP 2.0服务端资源推送。
  • https://github.com/fouber/blog/issues/6

Webpack

  1. webpack这个前端自动化构建工具，可以完美实现资源的合并、打包、压缩、混淆等诸多功能

  2. 使用nodejs开发出来的一个前端工具，写webpack.config.js这个配置文件，应该用commonjs规范来写

  3. 开发流程演变

    书写html文件---> 右键打开html文件 （js，css，图片，字体等都放在各种目录下面）
不利于有些资源的获取， 不能模拟现实的http请求的 各种状况，文件协议很多有不准确； 所以就有了开发服务器的说法

https://www.webpackjs.com/
现在我们使用webpack 对es6的 文件进行 转换为浏览器所支持的低版本 js
npm install webpack -g  这个是全局安装webpack， 在家目录下面的 AppData/Roaming/npm下面，属于一个命令行工具
npm install webpack  这个是安装在项目里面，在package.json中查看版本，属于开发依赖

  4. webpack的四个核心概念

    1. 入口(entry)
    2. 输出(output)
    3. loader
    4. 插件(plugins)
概念 具体内容
安装webpack npm install webpack -g
npm install webpack-cli -g 用于命令行
npm install webpack --save-dev 将webpack安装到 项目开发依赖
命令行工具使用 webpack app.js dist/bundle.js 将高级js转换为
html页面更改 index.html中引入的js文件 换位bundle.js
默认配置文件 使用打包工具的时候，默认的配置文件，打包入口文件名是 main.js
webpack 配置文件,核心
1. 入口和输出
var path = require('path');

// 就把他看做在写nodejs， 我们是使用nodejs周边，来辅助我们开发前端
// 导出一个配置对象，将来webpack在启动的时候，会默认来查找webpack.config.js
module.exports = {
  entry: path.resolve(__dirname, 'app.js'),  // 默认入口文件是main.js
  output: {
    path: path.resolve(__dirname, 'dist'),
    filename: 'bundle.js',
  },
};

在命令行输入 webpack，如果命令行参数没有指定入口和出口， 就会自动查找当前目录的webpack.config.js文件，解析对应配置文件，然后进行打包和构建
2. 开发服务器和插件的使用
1. 开发服务器的好处就是  自动实现代码实时打包编译，当修改代码之后，会自动进行打包构建； 不用我们手动使用webpack； 相当于上面哪一步就省略了

2. npm install  webpack-dev-server --save-dev  开发服务器的库
不要在全局里面直接输入webpack-dev-server； 利用script 节点里面配置任务； 调用当前项目的webpack-dev-server这个库； 要想在命令行直接使用，必须按照全局的这个工具 -g

3. 关于配置文件里面，开发服务器的一些配置是

devServer: {
    open: true, // 自动打开浏览器
    port: 8080, // 设置启动时候的运行端口
    hot: true, // 启用热更新,  ---- 有哪几个步骤呢
    compress: true, // 开启gzip压缩
},
一定注意， 这一步并没有在dist目录里面生成 bundle.js文件， 是放在内存中，这也是为什么我们开发非常快的原因，也可以叫热更新，

4. npm install html-webpack-plugin --save-dev   安装插件
每次编译之后，还要去修改index首页的 script， src属性，比较麻烦，所以有一种叫插件的东西，自动做这些事情，这个插件有两个作用
4.1  自动在内存中根据指定页面生成一个内存的页面
4.2  自动，把打包好的 bundle.js 追加到页面中去
3. loader的基本使用
1. loader可以对模块的源代码进行转换，webpack只识别低版本的js， 对于图片，字体图标，css等都要转换为模块； 有点类似于gulp里面的任务task
2. 我们来试着处理一下 项目中的 css，less，scss三种文件

安装包：
npm i style-loader css-loader less-loader less --save-dev
npm i sass-loader node-sass --D  这个sass在亚马逊云上面，不是很好安装，需要一些配置，或者使用cnpm单独安装
npm i url-loader file-loader -D  处理css中的路径


注意点：
1. loader的解析 都是从右往左，把做右边的结构交给坐标相邻的loader处理
2. sass 2007年处理，ruby编写；  less 2009年开源，受sass影响较大； 一些细微差别，如变量$ @
3. url-loader 通过一些配置，可以让图片在一定大小的时候转换为base64格式，进行项目优化，打包进bundle.js； 大于这个临界值的通过file-loader


module: {
  rules: [
    { test: /\.css$/, use: ['style-loader', 'css-loader'] },
    { test: /\.less$/, use: ['style-loader', 'css-loader', 'less-loader'] },
    { test: /\.scss$/, use: ['style-loader', 'css-loader', 'sass-loader'] },
    // 可以通过`limit`指定进行base64编码的图片大小；只有小于指定字节（byte）的图片才会进行base64编码
    // { test: /\.(png|jpg|jpeg|gif)$/, use: ['url-loader'] },
    { test: /\.(png|jpg|jpeg|gif)$/, use: 'url-loader?limit=8192&name=[name]-[ext].[hash:8]' },
  ],
},
vue里面webpack相关
module.exports = {
  configureWebpack: {
    module: {
      rules: [
      // 可以自己定义一些规则， 这里是小于1KB的就 不用进行base64编码，直接使用原始图片
        { test: /\.(png|jpg|jpeg|gif)$/, use: 'url-loader?limit=1024&name=[name]-[ext].[hash:8]' },
      ],
    },
  },
  devServer: {
    open: true, // 自动打开浏览器
    port: 9000, // 设置启动时候的运行端口
    hot: true, // 启用热更新,  ---- 有哪几个步骤呢
    compress: true, // 开启gzip压缩
  },
};
